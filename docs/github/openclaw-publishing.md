# OpenClaw GitHub Publishing Audit

## Цель

Подготовить `/Users/dmitriy/openclaw` к публикации как отдельного GitHub-репозитория с акцентом на нашу собственную работу, без случайной утечки локальных секретов и без путаницы с внешними источниками.

## Что публикуем

- наш workspace и локальные наработки вокруг OpenClaw;
- документацию, исследования, описания инструментов и организационные материалы;
- код и скрипты, которые действительно относятся к нашему процессу и могут быть воспроизведены без приватных данных;
- заготовки GitHub-материалов, скриншотов и описаний.

## Что не публикуем

- любые `.env`, ключи, токены, пароли, бинарные секреты и локальные учетные данные;
- `node_modules`, build-артефакты, логи, временные базы, служебные кэши;
- вложенные git-репозитории и чужие рабочие деревья;
- runtime-данные Telegram/trading, которые накапливаются в локальной эксплуатации;
- приватные скриншоты и реальные медиа, пока они не будут вручную отобраны.

## Отдельные решения

### `infrastructure/`

- Публиковать как нашу собственную разработку в лоб не стоит.
- В GitHub-описании основного репозитория лучше явно указать, что `infrastructure/` локально использовался как внешняя база/источник.
- На странице репозитория стоит дать ссылку на оригинальный upstream-источник.
- Внутри текущего workspace можно оставить каталог для локальной работы, но в публичном описании подчеркнуть, что это не центральная часть нашей авторской публикации.

### `tools/trading/`

- Имеет смысл вынести в отдельный публичный репозиторий.
- В его README нужно явно написать, что пакет создан для работы внутри/вокруг OpenClaw.
- На странице `tools/trading` и в описании отдельного репозитория стоит дать обратную ссылку на основной `openclaw` workspace.

## Найденные риски перед первым коммитом

### Высокий риск

- Найден локальный секрет в `core/backup/.env`. Этот файл нельзя публиковать ни при каких условиях.
- В workspace есть вложенные локальные репозитории, в частности `openclaw_x/.git`. Их нельзя тащить в основной публичный репозиторий без отдельного решения.

### Средний риск

- В дереве много `node_modules`, `dist`, `build`, логов и generated output.
- В trading-инструментах есть файлы runtime-истории и feedback (`*.jsonl`, `state.json`, публикационные журналы), которые лучше не коммитить в публичную версию.

### Низкий риск, но требует внимания

- В `infrastructure/` лежит крупный внешний код, который визуально может выглядеть как часть нашей авторской работы, если это не пояснить в README.
- Для публичной страницы понадобятся отобранные реальные изображения и скриншоты, а не весь сырой медианабор.

## Рекомендуемая схема публикации

1. Основной GitHub-репозиторий: наш workspace OpenClaw, без `infrastructure` как центрального claim и без локальных runtime-артефактов.
2. Отдельный GitHub-репозиторий: `tools/trading`, с явной ссылкой на OpenClaw как платформу/среду, для которой этот набор создавался.
3. Для обоих репозиториев: отдельные обложки, скриншоты интерфейса, скрины Telegram-канала и короткие описания в одном стиле.

## Перед публикацией вручную проверить

- все ли реальные скриншоты очищены от чувствительных данных;
- не попали ли в staged files локальные `.env`, `.jsonl`, runtime state и media;
- корректно ли указан upstream/source для `infrastructure/`;
- устраивает ли нас список директорий, которые полностью исключены из первого публичного коммита.
